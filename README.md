•	Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?
The Gaming Room client is a company that wanted to develop a game that serves multiple platforms based on their current game Draw It or Lose It that was only available on an Android app. I was tasked with preparing a software design document and to begin developing the game application that met The Gaming Room’s software requirements. The requirements were as follows: 
•	A game will have the ability to have one or more teams involved.
•	Each team will have multiple players assigned to it.
•	Game and team names must be unique to allow users to check whether a name is in use when choosing a team name.
•	Only one instance of the game can exist in memory at any given time. This can be accomplished by creating unique identifiers for each instance of a game, team, or player.
•	The game must be web-based and accessible on multiple platforms
•	Each round of the game is timed, 1 minute per round
•	The image should render fully within 30 seconds
•	A team has 30 seconds to guess what the image is, if they do not answer in that time, the other teams have 15 seconds to give one answer each to solve. 

•	What did you do particularly well in developing this documentation?
When developing this documentation, I think I did particularly well with outlining the needs of The Gaming Room and with explaining the advantages and disadvantages of using Linux, Mac, Windows, and Mobile Devices within the client and server side. 
•	What about the process of working through a design document did you find helpful when developing the code?
I found it helpful to list out all the requirements that the client wanted for the design of the game. When I know exactly what the client needs from the project, this really helps me with developing a plan for developing the code.
•	If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?
If I could choose one part of my work on these documents to revise, I would revise the recommendations portion of the document. I feel like I could have added more information outlining why I made certain recommendations and what those recommendations mean for the company’s program. 
•	How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?
It is important to figure out what the user needs for the software vs what they want from the software. The user’s needs are things that are required for the software to work properly and cannot be without. To interpret the user's needs, I have to pay attention to all the requirements that the user tells me and find the items that the software could not possibly be without and make sure that those items are properly implemented into the design. 
•	How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?
I made sure to look over all of the requirements that were expected of me and broke down each requirement into small pieces and slowly worked through each portion, always checking that things were done properly before moving forward. In the future, I would like to try using pseudocode more, as this is very helpful when coming up with a plan for software development.
